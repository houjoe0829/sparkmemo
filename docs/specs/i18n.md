# 国际化

## 目标与范围

所有面向用户的字符串走一层轻量翻译层，跟随 Obsidian 的语言设置在中文 / 英文之间切换。规定命名空间约定与「加新 key」的流程，避免代码里散落硬编码字面量。

**包含：**
- 语言检测策略
- 字典结构与命名空间约定
- `t()` / `currentLocale()` 的使用规范
- 加新 key 的流程

**不包含：**
- 长文本 / 长文案的翻译工作流（当前规模不需要）

## 用户可感知行为

- Obsidian 语言设为中文时，插件所有 UI（设置页、Notice toast、侧栏四个 Tab）都显示中文。
- 切到英文时，同样位置显示英文。
- 无第三种语言时按英文兜底。
- 数字 / 日期格式跟随语言（如 `1,234` vs `1234`），由调用方基于 `currentLocale()` 决定。

## 实现要点

- **翻译层**：[i18n.ts](../../src/i18n.ts) 定义 `dictionaries.en` / `dictionaries.zh` 两份字典，`t(key, vars?)` 按当前 locale 查表并做 `{var}` 替换。
- **locale 检测**：
  1. 优先 Obsidian 官方 `getLanguage()`（Obsidian ≥ 1.8.7）
  2. 回退 `moment.locale()`
  3. 兜底 `en`
- **命名空间**：字典按点分命名空间组织，各 Tab / 模块自己一段前缀：
  - `capture.*` — 记录 Tab
  - `search.*` — 搜索 Tab
  - `location.*` — 地点 Tab
  - `stats.*` — 统计 Tab
  - `notice.*` — Notice toast 文案
  - `settings.*` — 设置页 name/desc
- **`currentLocale()`**：给需要分支的调用方（比如日期 / 数字格式化）用，返回 `'zh'` / `'en'`。
- **调用位置**：所有 UI 文案都走 `t()`，包括 [notice.ts](../../src/notice.ts) 里的 toast 封装。

## 加新 key 的流程

1. 想清楚归属哪个命名空间（不确定就归 `notice.*` 或对应 Tab）
2. 在 [i18n.ts](../../src/i18n.ts) 的 `dictionaries.en` 和 `dictionaries.zh` 里各加一条，两边必须同时有
3. 在代码里用 `t('capture.your_key', { ...vars })` 调用；含变量时用 `{name}` 占位
4. 不要保留孤儿 key（只有单语言）

## 已知约束与遗留

- **只有中英**：暂不做第三语言；如果加，字典结构支持，locale 检测里加分支即可。
- **无提取工具**：字典手写，没有从代码自动提 key 的工具；靠 code review 保证不漏。
- **无复数规则**：字典不区分单复数，靠 key 本身或 `vars` 拼接；对中英够用，做多语言时再补。
- **老 CSS 类名**：`jp-` 前缀是内部命名，不进字典。
